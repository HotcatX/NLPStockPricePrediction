# 服务器配置与部署指南

## 1. 当前 AWS 资源状态
您目前提供的凭证列表中，包含以下资源：
*   **IAM 凭证** (Access Key, Secret Key): 用于验证身份。
*   **DynamoDB**: 云端 NoSQL 数据库，目前有表 `APAN5200`。
*   **Aurora MySQL**: 云端关系型数据库，具有特定的 Endpoint。*注意：您的 README 目前写的是 Aurora PostgreSQL，而实际凭证提供的是 MySQL。这可以通过修改代码或 README 解决，MySQL 同样适用于数据存放的任务。*
*   **Secrets Manager**: 存储 Aurora 密码。

**结论**：这些都是**数据服务 (Data Services)** 或 **托管数据库**，它们本身是用来存数据的，**不能用来运行和存放 Python 脚本**。

## 2. 如何存放和运行脚本 (计算资源 Compute)
要在云端存放和定时运行您的 Python 脚本（数据收集器），您需要在 AWS 上开通一种新的资源，最常见且符合学生免费额度的是 **Amazon EC2 (Elastic Compute Cloud)**。

就相当于在云端租用了一台有一块硬盘（EBS）和操作系统的虚拟机。

### 部署在 EC2 的基本操作：
1.  启动一台 EC2 实例 (通常选择 Amazon Linux 2023 或 Ubuntu，实例类型 `t2.micro` 或 `t3.micro` 可享受免费套餐)。
2.  通过 SSH 连接到这台服务器。
3.  像在本地电脑上一样，克隆 Github 代码仓库。
4.  配置 `.env` 文件，填入保存在本地私密说明文件中的认证信息。
5.  设置服务器上的定时任务 (例如 Linux 的 `cron`)，每天自动运行 `python data_loader.py`。
6.  (可选) 利用 EC2 通过 Flask 提供最终的数据展示面板的公网 IP。

## 3. 另一种轻量级选择：AWS Lambda (Serverless)
如果您不想维护一整台 EC2 云端电脑，也可以选择 **AWS Lambda**。
*   特点：只需上传 `data_loader.py` 代码，配置好执行环境。它不需要有常开的实例，可以利用 Amazon EventBridge 设置事件，设定在特定时间触发这段代码。由于按运行时的毫秒数计费，抓取点数据可能基本免费。

## 4. 后续方案同步修改建议
由于您提供的 Aurora 类型是 `MySQL` ，请将 `README.md` 与分析报告里的设计架构从 `PostgreSQL` 更新为 `MySQL` 以保持项目真实性与统一性。
