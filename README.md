# APAN5400 期末项目说明

## 一、项目基本信息

- 截止时间：5 月 3 日 11:59pm
- 总分：400 分
- 提交方式：文件上传
- 组队要求：小组作业，每组最多 5 人

期末项目需要围绕一个自选的大数据集展开，结合课程每周讲授内容、平时作业和测验中学到的方法，设计一个完整的数据分析应用场景，并实现端到端的数据处理系统。项目必须使用至少 3 种课程中学习过的技术。

一个典型示例是：设计一个信息检索或推荐系统，前端提供 API 或交互界面，后端同时结合 SQL 与 NoSQL 数据存储，并配合 ETL、分析处理或大数据框架完成整体方案。

## 二、正式交付内容

期末项目需要提交以下两部分内容：

- 数据存储设计与 ETL 流程设计
- 使用 Python 实现的端到端数据管理应用
- 现场演示与最终汇报

最终汇报需要包含：

- 数据分析问题的背景介绍与问题定义
- 数据来源说明，以及数据获取方式
- 系统设计方案、所选技术以及选择这些技术的理由
- 数据质量、许可协议、可扩展性与成本等方面的说明
- 结论与未来改进建议

在学期最后两次课程中，每个小组都需要进行展示，且每位组员都必须参与。

## 三、根据 PPT 模板整理的汇报要求

下面内容根据 `/Users/cat/Downloads/Managing Data Final Project Template (1).pptx` 提炼整理，建议直接作为最终 PPT 和 README 的主线结构。

### 1. 项目背景

需要说明：

- 你的项目要解决什么问题，或者抓住什么机会
- 项目的核心概念是什么
- 系统有哪些关键功能
- 为什么这是一个基于 Big Data 的项目

这一部分不能只写一句概念，必须讲清楚业务场景、目标用户和项目价值。

### 2. 数据来源与数据体量

需要说明：

- 项目会使用哪些数据源，且要尽量具体
- 数据源可能包括用户画像、人口统计、地理信息、用户交互、支付记录、开放平台 API、社区网站数据等
- 数据体量大概有多少，是否满足至少 1GB 的要求

模板还强调，演示部分建议至少覆盖以下技术形态：

- 一个 API
- 一个 SQL 数据库
- 一个 NoSQL 数据库
- 一个其他大数据工具，例如 Spark

### 3. 数据架构

模板要求展示完整的数据架构，至少要考虑下列方面：

- 数据采集工具
- 数据存储工具，例如 MongoDB、PostgreSQL
- 数据转换工具
- 实时流处理工具，例如 Kafka
- 商业智能工具
- 数据处理与分析工具，例如 Spark
- 云平台工具
- 数据治理工具
- Hadoop 或 Spark 生态相关工具
- AI 工具（如果适用）

这部分建议画成系统架构图或数据流图，而不是只列名词。

### 4. 架构工具的优缺点

模板特别要求你能解释所选工具的优缺点，至少要讲清楚以下几类：

- SQL 工具为什么要选它
- NoSQL 工具为什么要选它
- Big Data 工具为什么要选它
- 如果用了云平台，也要说明云平台方案的利弊

换句话说，不能只说“用了什么”，还要说明“为什么不用别的”。

### 5. 可扩展性与成本

需要说明：

- 如果项目从 NYC 扩展到更大范围，系统要怎么调整
- 当前技术栈在哪些地方会成为瓶颈
- 系统扩容后，计算、存储、调用 API、部署等成本会怎么变化

模板明确提醒：某些课堂中的轻量方案，例如 Flask 加简单 SQL，并不能天然支持大规模扩展，因此这里必须讨论扩展后的技术演进路线。

### 6. 数据治理、质量与合规

这一部分建议至少包含：

- 应用本身的安全性
- 数据架构工具的安全性
- 数据质量问题
- 数据授权与许可证问题
- 相关法律法规或行业规范

模板给出的例子是：如果涉及医疗信息，就需要考虑 HIPAA 等合规要求。你们应结合自己的数据领域，说明对应的合规风险。

### 7. 完整的数据湖或平台总图

模板明确要求最终项目给出一张完整图，综合展示你们提到的所有工具。

这一页应该体现：

- 数据从哪里来
- 如何被采集
- 如何进入数据湖、数据库或缓存
- 如何被清洗、转换、分析
- 如何服务于 API、前端或最终用户

如果你们的项目不是严格意义上的 Data Lake，也至少要给出一张完整的端到端系统总图。

### 8. Demo 要求

模板强调，Demo 的目标是展示本学期学到的技术如何落地，不要求做成完美商业产品，但必须“能展示、能解释、能支撑方案”。

建议在 Demo 中体现：

- 一个可见的界面或 API
- 一个真实可查询的 SQL 数据库
- 一个真实可查询的 NoSQL 数据库
- 至少一个大数据处理组件，例如 Spark
- 一段能体现 ETL、分析或推荐逻辑的 Python 代码

也就是说，Demo 不只是放代码截图，而是要让老师看到系统的实际运行逻辑。

### 9. 展示与答辩要求

模板中对展示提出了明确要求：

- 每位组员都要有实质性贡献
- 每位组员都应参与汇报
- 每位同学应大致了解整个项目，而不仅仅是自己负责的部分
- 小组展示时间大约为 5 分钟，之后还有大约 5 分钟 Q&A

因此，README 和 PPT 最好提前对每位成员负责的模块进行分工说明。

### 10. 最终交付提醒

模板还补充说明：

- Demo 不需要做到完美，但在最终展示日必须达到可演示、可讲解的程度
- 最终代码通常会在展示日后再给出 1 到 2 天补交空间，但不能依赖这个缓冲
- 课程会在后期留出专门时间供小组推进项目

这意味着你们应该优先保证“演示闭环”先完成，再继续打磨附加功能。
